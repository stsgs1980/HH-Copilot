# Архитектура Telegram-бота для автоматического отклика на вакансии HH.ru

---

## 1. Обзор решения

Данный документ описывает архитектуру Telegram-бота, обеспечивающего автоматический поиск и отклик на релевантные вакансии с платформы HH.ru. Бот выполняет авторизацию пользователя через OAuth 2.0, загружает и парсит резюме, ищет вакансии с оценкой релевантности не менее 70%, автоматически откликается от имени пользователя и предоставляет расширенные возможности: смену карьерного направления, генерацию сопроводительных писем и автоматические/ручные ответы работодателям.

Решение строится на микросервисной архитектуре с выделенными модулями для каждого функционального слоя: авторизация, парсинг резюме, поиск и ранжирование вакансий, генерация текстов на базе LLM, отклики и переписка. В качестве основного оркестратора выступает Telegram-бот на базе `aiogram`, а фоновые задачи (периодический поиск, мониторинг откликов) обрабатываются через `Celery` + `Redis`.

## 3. Детальное описание компонентов

### 3.1. Telegram Bot Layer (aiogram 3.x)

Слой взаимодействия с пользователем через Telegram. Реализован на базе фреймворка `aiogram 3.x` с использованием FSM (Finite State Machine) для управления диалогами. Каждый хэндлер отвечает за свою предметную область и делегирует бизнес-логику в Core Service Layer.

| Компонент | Назначение | Ключевые команды |
|-----------|-----------|-----------------|
| Auth Handler | Авторизация через OAuth 2.0 | `/start`, `/login`, `/logout` |
| Resume Handler | Управление резюме | `/resumes`, `/parse`, `/select_resume` |
| Search Handler | Поиск и фильтрация вакансий | `/search`, `/filters`, `/direction` |
| Negotiation Handler | Отклики и переписка | `/apply`, `/auto_apply`, `/messages`, `/reply` |

**Auth Handler** инициирует OAuth-процесс, генерируя уникальную ссылку для авторизации на HH.ru. После успешного редиректа получает `authorization_code`, обменивает его на `access_token` и `refresh_token`, после чего сохраняет токены в БД с привязкой к `telegram_user_id`. При истечении `access_token` автоматически использует `refresh_token` для продления сессии.

**Resume Handler** позволяет пользователю просмотреть список своих резюме, выбрать одно или несколько для работы, а также инициировать глубокий парсинг резюме для извлечения ключевых компетенций, навыков, опыта и образования. Выбранные резюме используются как профиль для оценки релевантности вакансий.

**Search Handler** предоставляет интерфейс для настройки параметров поиска: регион, зарплата, опыт, тип занятости, график работы, ключевые слова. Отдельно поддерживается команда `/direction` для смены карьерного направления — пользователь указывает новую сферу или должность, и это направление фиксируется как дополнительный критерий фильтрации поверх стандартных фильтров HH.ru.

**Negotiation Handler** управляет откликами и перепиской. Поддерживает два режима: автоматический (бот сам откликается на вакансии с релевантностью >= 70%) и ручной (пользователь подтверждает каждый отклик). Также обеспечивает чтение и отправку сообщений работодателям с опциями автоответа.

### 3.2. Core Service Layer

Ядро бизнес-логики, независимое от транспортного слоя (Telegram). Каждый сервис инкапсулирует свою доменную область и взаимодействует с HH.ru API через единый API Client.

#### 3.2.1. Auth Service

Отвечает за полный цикл OAuth 2.0 авторизации: генерация авторизационной ссылки, обмен кода на токен, обновление токена, хранение и валидация. Использует шифрование для хранения токенов в БД (AES-256-GCM). При каждом запросе к HH.ru API проверяет валидность токена и при необходимости прозрачно обновляет его.

**Ключевые методы:**
- `get_auth_url(telegram_user_id)` — генерирует URL для авторизации с `state`-параметром
- `exchange_code(code, state)` — обменивает `authorization_code` на токены
- `refresh_access_token(user_id)` — обновляет `access_token` через `refresh_token`
- `get_valid_token(user_id)` — возвращает действительный токен (с автоматическим refresh)
- `revoke_token(user_id)` — отзывает токен при выходе

#### 3.2.2. Resume Parser

Извлекает и структурирует данные из резюме пользователя для последующего использования в оценке релевантности и генерации сопроводительных писем. Парсер работает с полным набором полей резюме, предоставляемых HH.ru API, и формирует внутреннее представление профиля.

**Извлекаемые данные:**
- Ключевые навыки (skill_set) — список строк
- Опыт работы (experience) — компании, должности, описания, периоды
- Образование (education) — учреждения, специальности, года
- Знание языков (language) — язык и уровень владения
- Профобласть и специализация (specialization)
- Должность (title) — текущая желаемая должность
- О себе (about) — свободное текстовое описание
- Квалификации и сертификаты

**Методы:**
- `fetch_user_resumes(user_id)` — получает список резюме через API
- `parse_resume(resume_id)` — загружает и детально парсит резюме
- `build_profile(resume_ids)` — формирует агрегированный профиль из нескольких резюме
- `extract_skills(profile)` — извлекает и нормализует навыки
- `get_career_vector(profile)` — вычисляет вектор карьерного направления на основе опыта

#### 3.2.3. Matching Engine

Центральный компонент системы, выполняющий оценку релевантности вакансии профилю пользователя. Использует гибридный подход: комбинацию семантического сравнения (через эмбеддинги) и структурного matching (по ключевым навыкам, опыту, образованию).

**Алгоритм оценки релевантности (0-100%):**

| Критерий | Вес | Описание |
|----------|-----|----------|
| Семантическое сходство (embedding) | 30% | Косинусное сходство между эмбеддингами профиля и вакансии |
| Совпадение ключевых навыков | 25% | Процент навыков вакансии, присутствующих в профиле |
| Соответствие опыта | 20% | Совпадение требуемого опыта с опытом кандидата |
| Соответствие должности | 15% | Семантическое сходство желаемой должности и названия вакансии |
| Образование и квалификации | 10% | Совпадение уровня образования и релевантных квалификаций |

**Порог релевантности:** >= 70% для автоматического отклика.

**Методы:**
- `compute_relevance(profile, vacancy)` — вычисляет итоговый скор релевантности
- `batch_rank(profile, vacancies)` — ранжирует список вакансий по релевантности
- `semantic_similarity(text_a, text_b)` — косинусное сходство эмбеддингов
- `skills_overlap(profile_skills, vacancy_skills)` — процент пересечения навыков
- `apply_direction_filter(vacancies, direction)` — фильтрация по карьерному направлению

**Хранение эмбеддингов:** Векторная БД Qdrant используется для хранения эмбеддингов резюме и вакансий, что позволяет быстро вычислять семантическое сходство через approximate nearest neighbor (ANN) поиск.

#### 3.2.4. Negotiation Service

Управляет полным циклом откликов и переписки с работодателями: создание отклика, мониторинг статусов, чтение и отправка сообщений.

**Режимы работы:**
- **Автоматический** — бот откликается на вакансии с релевантностью >= 70% без подтверждения пользователя
- **Полуавтоматический** — бот предлагает вакансию, пользователь подтверждает или отклоняет
- **Ручной** — пользователь сам инициирует отклик на конкретную вакансию

**Методы:**
- `apply_to_vacancy(user_id, vacancy_id, resume_id, cover_letter)` — отклик на вакансию
- `get_negotiations(user_id, status)` — получение списка откликов
- `get_messages(user_id, negotiation_id)` — чтение переписки
- `send_message(user_id, negotiation_id, text)` — отправка сообщения
- `auto_reply(user_id, negotiation_id, employer_message)` — генерация автоответа через LLM

### 3.3. HH.ru API Client

Единый клиент для взаимодействия с HH.ru API. Обрабатывает аутентификацию, rate limiting, повторные запросы при ошибках и десериализацию ответов. Все запросы проходят через этот клиент, что обеспечивает централизованный контроль и логирование.

**Конфигурация:**
- Базовый URL: `https://api.hh.ru`
- Обязательный заголовок: `User-Agent: AppName/Version (contact@email.com)`
- Авторизация: `Authorization: Bearer {access_token}`
- Формат: JSON
- Rate limit: 30 запросов в минуту для авторизованных пользователей (обрабатывается автоматически через экспоненциальный backoff)

### 3.4. Background Workers (Celery)

Фоновые задачи, выполняемые периодически через Celery Beat:

| Задача | Расписание | Описание |
|--------|-----------|----------|
| `search_new_vacancies` | Каждые 30 мин | Поиск новых вакансий по фильтрам пользователя |
| `monitor_negotiations` | Каждые 5 мин | Проверка новых сообщений от работодателей |
| `auto_apply_batch` | По требованию | Массовый отклик на подходящие вакансии |
| `refresh_tokens` | Каждый час | Упреждающее обновление токенов до истечения |
| `generate_cover_letters` | По требованию | Генерация сопроводительных писем через LLM |
| `update_vectors` | Каждые 6 ч | Обновление эмбеддингов профиля и вакансий |

### 3.5. Data Layer

| Хранилище | Назначение | Данные |
|-----------|-----------|--------|
| PostgreSQL | Основная БД | Профили пользователей, токены, настройки фильтров, история откликов, статусы |
| Redis | Кэш + очередь задач | Кэш ответов API, сессии FSM, задачи Celery, временные данные |
| S3/MinIO | Файловое хранилище | Логи, экспортируемые отчёты, резервные копии |
| Qdrant | Векторная БД | Эмбеддинги резюме, вакансий, навыков для семантического поиска |

## 4. Эндпоинты HH.ru API по шагам сценария

### 4.1. Шаг 1: Авторизация через OAuth 2.0

HH.ru использует стандартный протокол OAuth 2.0 (Authorization Code Grant). Для получения доступа к данным пользователя необходимо зарегистрировать приложение на HH.ru (dev.hh.ru), получить `client_id` и `client_secret`, после чего реализовать следующий поток:

#### 4.1.1. Перенаправление пользователя на страницу авторизации

```
GET https://hh.ru/oauth/authorize
```

| Параметр | Значение | Описание |
|----------|----------|----------|
| `response_type` | `code` | Тип OAuth-гранта |
| `client_id` | `{CLIENT_ID}` | Идентификатор приложения |
| `redirect_uri` | `{REDIRECT_URI}` | URL для возврата после авторизации |
| `state` | `{random_state}` | Защита от CSRF, привязан к `telegram_user_id` |

**Поток в боте:** Бот генерирует уникальный `state`, сохраняет привязку `state -> telegram_user_id` в Redis (с TTL 10 мин), формирует авторизационную ссылку и отправляет её пользователю в виде inline-кнопки. После авторизации на HH.ru пользователь перенаправляется на `redirect_uri` с параметром `code`.

#### 4.1.2. Обмен authorization_code на access_token

```
POST https://api.hh.ru/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&client_id={CLIENT_ID}
&client_secret={CLIENT_SECRET}
&code={AUTHORIZATION_CODE}
&redirect_uri={REDIRECT_URI}
```

**Ответ:**

```json
{
  "access_token": "abcdef123456",
  "refresh_token": "xyz789012345",
  "token_type": "Bearer",
  "expires_in": 1209600
}
```

`access_token` действует 14 дней (1 209 600 секунд). `refresh_token` позволяет обновить `access_token` без повторной авторизации пользователя.

#### 4.1.3. Обновление access_token

```
POST https://api.hh.ru/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token={REFRESH_TOKEN}
```

**Ответ:** аналогичен п. 4.1.2 с новыми токенами.

#### 4.1.4. Проверка токена и получение информации о пользователе

```
GET https://api.hh.ru/me
Authorization: Bearer {access_token}
```

Возвращает информацию об авторизованном соискателе: имя, email, статус и пр. Используется для валидации токена и первичного получения данных пользователя.

#### 4.1.5. Отзыв токена

```
DELETE https://api.hh.ru/token
Authorization: Bearer {access_token}
```

Используется при команде `/logout` для отзыва авторизации.

### 4.2. Шаг 2: Загрузка и парсинг резюме

#### 4.2.1. Получение списка резюме пользователя

```
GET https://api.hh.ru/resumes/mine
Authorization: Bearer {access_token}
```

Возвращает список всех резюме пользователя с краткой информацией по каждому. Требует авторизации, иначе вернёт `403 Forbidden`.

**Ключевые поля ответа:**
- `id` — идентификатор резюме (hash-строка)
- `title` — желаемая должность
- `status` — статус резюме (опубликовано, скрыто и т.д.)
- `updated_at` — дата последнего обновления
- `url` — ссылка на полное резюме

#### 4.2.2. Получение детальной информации по резюме

```
GET https://api.hh.ru/resumes/{resume_id}
Authorization: Bearer {access_token}
```

Возвращает полную информацию по указанному резюме. Требует авторизации владельца.

**Ключевые поля для парсинга:**

| Поле | Тип | Описание |
|------|-----|----------|
| `skill_set` | `string[]` | Список ключевых навыков |
| `experience` | `object[]` | Опыт работы: компания, должность, описание, период |
| `education` | `object[]` | Образование: учреждение, факультет, специальность |
| `language` | `object[]` | Знание языков с уровнем владения |
| `specialization` | `object[]` | Профобласти и специализации |
| `title` | `string` | Желаемая должность |
| `about` | `string` | О себе |
| `salary` | `object` | Желаемая зарплата и валюта |
| `area` | `object` | Город/регион |
| `employment` | `string[]` | Типы занятости |
| `schedule` | `string[]` | Графики работы |
| `citizenship` | `object[]` | Гражданство |
| `work_ticket` | `object[]` | Разрешение на работу |
| `certificate` | `object[]` | Сертификаты и квалификации |

#### 4.2.3. Дополнительные справочники для обогащения данных

```
GET https://api.hh.ru/dictionaries
```

Возвращает все справочники: типы занятости, графики работы, уровни опыта, валюты и т.д. Используется для корректной интерпретации кодов в данных резюме и вакансий.

```
GET https://api.hh.ru/specializations
```

Возвращает дерево профессиональных областей и специализаций. Используется для маппинга профиля на профобласти и для реализации функции смены карьерного направления.

### 4.3. Шаг 3: Автоматический поиск вакансий и оценка релевантности

#### 4.3.1. Поиск вакансий

```
GET https://api.hh.ru/vacancies
Authorization: Bearer {access_token}  (опционально)
```

**Параметры запроса:**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `text` | `string` | Поисковый запрос (поддерживает язык запросов HH.ru) |
| `search_field` | `string` | Область поиска: `name`, `company_name`, `description`. По умолчанию — все поля |
| `experience` | `string` | Опыт: `noExperience`, `between1And3`, `between3And6`, `moreThan6` |
| `employment` | `string` | Тип занятости: `full`, `part`, `project`, `volunteer`, `probation` |

| `currency` | `string` | Код валюты: `RUR`, `EUR`, `USD` и др. |

## 5. Дополнительный функционал

### 5.1. Смена карьерного направления

Пользователь может задать новую сферу или должность, которая фиксируется как дополнительный критерий фильтрации. Реализация включает несколько уровней:

**Уровень 1: Фильтрация через API HH.ru.** При указании нового направления бот маппит пользовательский ввод на `specialization` ID из справочника HH.ru и добавляет этот фильтр в параметры поиска. Также возможно использование параметра `text` для текстового поиска по названию должности.

**Уровень 2: Семантическая фильтрация.** Если пользовательское направление не маппится напрямую на справочник HH.ru (например, «хочу перейти из бухгалтерии в аналитику данных»), система генерирует вектор нового направления через LLM и эмбеддинги, затем использует его как дополнительный фильтр при ранжировании вакансий. Это позволяет находить вакансии, которые формально относятся к другой профобласти, но по сути соответствуют желаемому направлению.

**Уровень 3: Адаптация профиля.** LLM анализирует текущий опыт пользователя и формирует «мост перенавыков» — описание того, как существующие навыки применимы в новом направлении. Эта информация используется при генерации сопроводительных писем и автоответов, чтобы подчеркнуть релевантность кандидата несмотря на смену сферы.

**Пользовательский сценарий:**
1. Пользователь отправляет команду `/direction Аналитик данных`
2. Бот маппит направление на специализации HH.ru: «Аналитик», «Data Scientist», «BI-аналитик»
3. Бот обновляет фильтры поиска, добавляя `specialization=1.221` (ID аналитика) и `text=аналитик данных`
4. Matching Engine получает повышенный вес для семантического сходства с новым направлением
5. При генерации сопроводительных писем подчёркивается мотивация к смене направления и переносимые навыки

### 5.2. Генерация сопроводительных писем

Сопроводительные письма генерируются через LLM с учётом:
- Опыта и специфики прошлых должностей (из парсинга резюме)
- Требований конкретной вакансии (ключевые навыки, описание)
- Желаемого карьерного направления (если задано)
- Тона и стиля, настроенных пользователем (формальный/дружелюбный)

**Prompt-шаблон для LLM:**

```
Ты — карьерный консультант. Напиши сопроводительное письмо для отклика на вакансию.

Профиль кандидата:
- Должность: {current_title}
- Опыт: {experience_summary}
- Ключевые навыки: {skills}
- О себе: {about}

Вакансия:
- Название: {vacancy_name}
- Компания: {employer_name}
- Требования: {vacancy_description}
- Ключевые навыки: {vacancy_key_skills}

{career_direction_hint}

Требования к письму:
- Объём: 150-250 слов
- Тон: профессиональный, но не сухой
- Подчеркни релевантные навыки и опыт
- Укажи конкретные достижения, связанные с требованиями вакансии
```

Письмо генерируется для каждого отклика индивидуально, что обеспечивает высокую персонализацию и повышает шансы на приглашение.

### 5.3. Ответы работодателям с вариантами авто/ручного режима

Система поддерживает три режима ответов:

**Автоматический режим:**
- Бот мониторит новые сообщения от работодателей через периодический опрос `GET /negotiations/{nid}/messages`
- При получении нового сообщения LLM анализирует контекст (предыдущая переписка, вакансия, профиль кандидата)
- Генерируется релевантный ответ и автоматически отправляется через `POST /negotiations/{nid}/messages`
- Пользователь получает уведомление в Telegram с текстом ответа

**Полуавтоматический режим (по умолчанию):**
- Бот обнаруживает новое сообщение и генерирует черновик ответа через LLM
- Черновик отправляется пользователю в Telegram с inline-кнопками «Отправить» / «Редактировать» / «Пропустить»
- Пользователь может отредактировать текст перед отправкой или одобрить как есть

**Ручной режим:**
- Бот уведомляет о новом сообщении от работодателя
- Пользователь сам набирает ответ, который пересылается через API

**Настройка режима:** Команда `/settings replies` позволяет выбрать режим и настроить параметры: тон ответов (формальный/неформальный), максимальная длина, время задержки перед автоответом (для возможности отмены).

## 7. Модель данных (PostgreSQL)

### 7.1. Основные таблицы

```sql
-- Пользователи Telegram
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    telegram_id     BIGINT NOT NULL UNIQUE,
    username        VARCHAR(255),
    first_name      VARCHAR(255),
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- OAuth-токены HH.ru
CREATE TABLE hh_tokens (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    access_token    TEXT NOT NULL,        -- зашифрован AES-256-GCM
    refresh_token   TEXT NOT NULL,        -- зашифрован
    expires_at      TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Профили резюме (распарсенные данные)
CREATE TABLE resume_profiles (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    hh_resume_id    VARCHAR(255) NOT NULL,
    title           VARCHAR(500),
    skills          JSONB,                -- ["Python", "SQL", "FastAPI", ...]
    experience      JSONB,                -- [{company, position, description, period}, ...]
    education       JSONB,                -- [{institution, faculty, year}, ...]
    specializations JSONB,                -- [{id, name}, ...]
    about           TEXT,
    embedding_id    VARCHAR(255),         -- ID векторного представления в Qdrant
    parsed_at       TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(user_id, hh_resume_id)
);

-- Настройки поиска
CREATE TABLE search_settings (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE UNIQUE,
    text_query      VARCHAR(500),
    area_ids        INTEGER[],            -- ID регионов
    salary_min      INTEGER,
    currency        VARCHAR(10) DEFAULT 'RUR',
    experience_id   VARCHAR(50),          -- noExperience, between1And3, etc.
    employment_ids  VARCHAR(50)[],        -- full, part, project, etc.
    schedule_ids    VARCHAR(50)[],        -- fullDay, remote, etc.
    specialization_ids VARCHAR(50)[],     -- ID профобластей
    period_days     INTEGER DEFAULT 7,
    relevance_threshold FLOAT DEFAULT 0.7,
    auto_apply      BOOLEAN DEFAULT FALSE,
    reply_mode      VARCHAR(20) DEFAULT 'semi_auto',  -- auto, semi_auto, manual
    career_direction TEXT,                -- пользовательский ввод нового направления
    direction_embedding_id VARCHAR(255),  -- вектор направления в Qdrant
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Выбранные резюме для работы
CREATE TABLE selected_resumes (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    resume_profile_id BIGINT NOT NULL REFERENCES resume_profiles(id),
    selected_at     TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(user_id, resume_profile_id)
);

-- История откликов
CREATE TABLE application_log (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    vacancy_id      VARCHAR(255) NOT NULL,
    vacancy_name    VARCHAR(500),
    employer_name   VARCHAR(500),
    resume_id       VARCHAR(255) NOT NULL,
    relevance_score FLOAT NOT NULL,
    cover_letter    TEXT,
    negotiation_id  VARCHAR(255),              -- ID отклика на HH.ru
    status          VARCHAR(50) DEFAULT 'sent', -- sent, viewed, invited, rejected, hidden
    applied_at      TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(user_id, vacancy_id, resume_id)
);

-- Шаблоны сопроводительных писем
CREATE TABLE cover_letter_templates (
    id              BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name            VARCHAR(255) NOT NULL,
    template_text   TEXT NOT NULL,             -- с плейсхолдерами {{vacancy_name}}, {{employer}}, etc.
    tone            VARCHAR(20) DEFAULT 'professional',
    is_default      BOOLEAN DEFAULT FALSE,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
```

## 8. Потоки данных (Data Flow)

### 8.1. Поток авторизации

```
User → /login → Bot генерирует state → Redis: state↔telegram_id
    → Кнопка с OAuth URL → HH.ru /oauth/authorize
    → Пользователь авторизуется → Redirect на callback URL
    → FastAPI callback → обмен code на token → POST /token
    → Шифрование токенов → PostgreSQL: hh_tokens
    → Bot уведомляет об успешной авторизации
```

### 8.2. Поток парсинга резюме

```
User → /resumes → HH.ru API: GET /resumes/mine
    → Список резюме → Telegram: inline-кнопки выбора
    → User выбирает → HH.ru API: GET /resumes/{id} (для каждого выбранного)
    → Парсинг: skills, experience, education, specializations
    → Формирование агрегированного профиля
    → Генерация эмбеддинга профиля → Qdrant: upsert
    → PostgreSQL: resume_profiles
```

### 8.3. Поток поиска и оценки релевантности

```
Celery Beat (каждые 30 мин) → Чтение search_settings из PostgreSQL
    → Формирование запроса к HH.ru → GET /vacancies (с фильтрами)
    → Итерация по страницам (до 5 страниц × 100 вакансий)
    → Для каждой вакансии:
        → Проверка: уже откликались? (application_log)
        → GET /vacancies/{id} (детальная информация)
        → Matching Engine:
            → Косинусное сходство (Qdrant: search по эмбеддингу профиля)
            → Skills overlap (сравнение skill_set и key_skills)
            → Experience matching
            → Итоговый скор = взвешенная сумма
        → Если скор >= порога:
            → auto_apply=True: немедленный отклик
            → auto_apply=False: добавление в очередь на подтверждение
    → Уведомление пользователя в Telegram
```

### 8.4. Поток отклика на вакансию

```
Matching Engine → вакансия с relevance >= 70%
    → LLM: генерация сопроводительного письма (cover_letter)
    → Проверка: response_letter_required? → если да, письмо обязательно
    → HH.ru API: POST /vacancies/{vacancy_id}/negotiations
        → resume_id из selected_resumes
        → message = cover_letter
    → Успех → PostgreSQL: application_log (status=sent)
    → Ошибка already_applied → логирование, пропуск
    → Уведомление пользователя: «Отклик отправлен на {vacancy_name}»
```

### 8.5. Поток мониторинга переписки

```
Celery Beat (каждые 5 мин) → Для каждого пользователя с активным токеном:
    → HH.ru API: GET /negotiations?status=active
    → Для каждого активного отклика:
        → GET /negotiations/{nid}/messages
        → Сравнение с последним обработанным сообщением (Redis: last_msg_{nid})
        → Если есть новые сообщения от работодателя:
            → Если reply_mode=auto:
                → LLM: генерация ответа (контекст: переписка + вакансия + профиль)
                → POST /negotiations/{nid}/messages
                → Уведомление: «Автоответ отправлен»
            → Если reply_mode=semi_auto:
                → LLM: генерация черновика
                → Telegram: сообщение с inline-кнопками [Отправить] [Редактировать] [Пропустить]
            → Если reply_mode=manual:
                → Telegram: уведомление о новом сообщении
    → Обновление last_msg в Redis
```

## 9. Ограничения и риски

### 9.1. Ограничения HH.ru API

| Риск | Описание | Митигация |
|------|----------|-----------|
| **Ограничение API для соискателей** | HH.ru последовательно ограничивает API-доступ для соискательских приложений, приоритизируя работодателей | Иметь fallback на веб-скрапинг (Playwright) для операций, недоступных через API; мониторинг изменений в документации API |
| **Премодерация приложений** | Новые приложения проходят ручную модерацию перед получением доступа к API | Заранее зарегистрировать и пройти модерацию; иметь описание приложения с обоснованием использования |
| **Rate limiting** | 30 запросов/мин для авторизованных пользователей | Экспоненциальный backoff, кэширование в Redis, пагинация с задержкой |
| **Отсутствие WebSocket** | Нет real-time уведомлений, только polling | Polling каждые 5 мин для переписки, каждые 30 мин для поиска вакансий |
| **Ограничение откликов** | HH.ru может лимитировать количество откликов в сутки | Квотирование откликов (не более 50/сутки), приоритизация по релевантности |

### 9.2. Технические риски

| Риск | Описание | Митигация |
|------|----------|-----------|
| **LLM-галлюцинации** | Сгенерированные сопроводительные письма могут содержать неточности | Человек в цикле (semi_auto по умолчанию); валидация ключевых фактов против данных резюме |
| **Безопасность токенов** | Утечка OAuth-токенов | Шифрование AES-256-GCM в БД; токены не логируются; минимальные scope |
| **Переполнение Qdrant** | Большой объём векторов при активном поиске | TTL для векторов вакансий (7 дней); периодическая очистка |
| **Зависимость от LLM** | Недоступность или высокая стоимость API | Fallback-шаблоны сопроводительных писем; кэширование типовых ответов |

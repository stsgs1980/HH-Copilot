# HH Copilot -- Chrome Extension

**Версия:** 1.5.2
**Тип:** Расширение для Chrome (Manifest V3)
**Целевая платформа:** hh.ru (Magritte дизайн-система)
**Лицензия:** Приватный проект. Все права защищены.


## 1. HH Copilot -- краткое описание

HH Copilot -- это браузерное расширение для автоматизации поиска работы на портале hh.ru. Проект решает задачу рутинного отклика на вакансии: парсит карточки вакансий со страницы поиска, извлекает данные резюме пользователя, показывает результаты во встроенной боковой панели и позволяет откликаться на подходящие позиции в один клик или в полностью автоматическом режиме.

Расширение работает исключительно в браузере пользователя. Не требуется серверная инфраструктура, база данных, Telegram-бот или какие-либо внешние API. Все данные хранятся локально в chrome.storage.local. Пользователь использует свою текущую сессию hh.ru, поэтому не нужна дополнительная авторизация через OAuth, OTP или 2FA. Расширение читает DOM-дерево страницы и взаимодействует с элементами так же, как это делал бы живой пользователь, что делает его неотличимым от ручных действий.

Проект изначально разрабатывался как часть большой системы автоматизации (Next.js + FastAPI + Playwright + Celery + Telegram), однако закрытие Applicant API hh.ru в декабре 2025 года сделало серверный подход неработоспособным. Playwright детектировался anti-bot системой по TLS-отпечатку, navigator.webdriver и поведенческой биометрии. Расширение решает все эти проблемы: оно работает в настоящем браузере, использует настоящие cookies, настоящие клики и настоящую навигацию. Никаких скрытых маркеров, никаких headless-сессий, никаких подмен user-agent.

Целевая аудитория -- соискатели, которые рассылают десятки откликов в день и хотят ускорить этот процесс без потери качества. Расширение не заменяет человека полностью, а автоматизирует механическую часть: поиск, фильтрацию, заполнение полей, отправку отклика. Решение о том, на какие вакансии откликаться, принимает пользователь (ручной режим) или алгоритм скоринга (полуавтоматический и автоматический режимы).

Текущая версия (1.5.0) поддерживает полный цикл парсинга резюме -- 12 из 12 полей извлекаются корректно на страницах с Magrite-вёрсткой. Парсинг вакансий со страницы поиска, список резюме со страницы /applicant/resumes, FAB-кнопка, Shadow DOM sidebar, MutationObserver для SPA-навигации и проверка авторизации -- всё это работает стабильно.


## 2. Возможности

### Что умеет расширение сейчас

**Парсинг вакансий со страницы поиска.** Расширение находит все карточки вакансий на странице hh.ru/search/vacancy и извлекает из каждой: название позиции, название компании, зарплату, локацию, требуемый опыт, теги навыков, URL и идентификатор. Данные проходят валидацию (проверка названия, компании, URL, id), фильтруются по чёрному списку компаний и списку уже откликнутых вакансий.

**Парсинг резюме (12 из 12 полей).** При открытии страницы конкретного резюме (/resume/{hash}) расширение извлекает: должность, желаемую зарплату, пол, возраст, город, навыки с уровнями владения (Продвинутый/Средний/Начальный), опыт работы (компания, позиция, период, описание), образование (название, год окончания), языки, дополнительную информацию. Для каждого поля есть fallback-стратегия: сначала data-qa селекторы, затем BEM-классы Bloko, затем текстовый анализ содержимого. Встроенный диагностический инструмент diagnoseResumeDOM() выводит полный дамп DOM в консоль для подбора селекторов.

**Список резюме.** На странице /applicant/resumes расширение находит все резюме пользователя и показывает их в sidebar с кликабельными ссылками. Уже загруженные резюме помечаются бейджем "loaded".

**Встроенная панель (FAB + Sidebar).** Плавающая кнопка (FAB) в правом нижнем углу экрана имеет три состояния: серая (проверка авторизации), красная (не авторизован), синяя (авторизован). При клике открывается боковая панель шириной 360px с табами "Вакансии" и "Моё резюме". Панель изолирована через Shadow DOM (mode: closed), поэтому стили hh.ru не влияют на её отображение, а стили панели -- на hh.ru.

**Проверка авторизации.** 13 CSS-селекторов + fallback по cookies (hhruuid, _HH-RU, hhtoken). Поллинг каждые 2 секунды. Определяет наличие активной сессии без запроса дополнительных разрешений.

**SPA-навигация.** hh.ru -- Single Page Application на React. При переходе на следующую страницу поиска URL меняется, но страница не перезагружается. MutationObserver с debounce в 1 секунду отслеживает изменения в DOM и автоматически перепарсивает вакансии.

**Popup-дашборд.** Четыре вкладки: статистика (отклики сегодня, приглашения, ошибки, всего), настройки (режим работы, лимиты, интервалы, отображение), шаблоны сопроводительного письма (с переменными {name}, {position}, {company}, {skills}), логи (последние 50 записей).

**Anti-hallucination.** Трёхуровневая система верификации: DOM Verification (проверка существования и видимости элемента), Data Validation (проверка типов и форматов данных), Action Verification (проверка предусловий и результатов действий). Все функции возвращают конкретные типы (string, null, boolean, number), никогда undefined.

**Rate Limiter.** Token bucket с адаптивным замедлением: 200 откликов/день, 30/час, минимум 30 секунд между действиями, максимум 5 подряд с паузой 2 минуты. При получении 429/капчи/замедленного ответа адаптивный фактор увеличивается (x2 при 429, x1.5 при slow, x1.3 при captcha).

### Что планируется

**Парсинг вакансий (детальный).** Переход на страницу конкретной вакансии и извлечение полного описания, требований, условий, навыков из текста вакансии.

**Matching-движок.** Пятикомпонентный скоринг: навыки (Jaccard + alias matching, 30%), зарплата (overlap-based, 25%), опыт (range parsing, 20%), позиция (word overlap, 15%), локация (exact/remote, 10%). Цветовая кодировка в sidebar: зелёный при score >= 70, жёлтый при 40-69, красный при < 40.

**Автоматический отклик.** Восьмишаговый процесс: проверка лимитов, навигация на страницу вакансии, поиск кнопки отклика, верификация текста кнопки, клик, ожидание popup, обработка алертов (релокация, тестовое задание, посредник), заполнение сопроводительного письма (имитация набора посимвольно), отправка и верификация результата.

**AI-сопроводительные письма.** Генерация персонализированных писем через LLM API с fallback на шаблон. Промпты уже подготовлены в docs/reference-prompts.py.

**Переговоры.** Отслеживание сообщений от работодателей в разделе переговоров и автоответ по шаблонам.

**Аналитика.** Графики откликов по дням/неделям, skill gap анализ (какие навыки востребованы, но отсутствуют в резюме).


## 3. Установка и запуск

### Сборка из исходников

Расширение не требует сборки. Это набор статических файлов: один JavaScript-бандл (content.js), один Service Worker (background/index.js), один HTML-файл popup с инлайн-стилями и скриптом, три PNG-иконки и конфигурация manifest.json. Никакие бандлеры, транспайлеры или сборщики не нужны. Всё работает в нативном виде в Chrome.

Для начала работы достаточно клонировать репозиторий и загрузить папку расширения в Chrome:

```
git clone https://github.com/stsgs1980/HH-Copilot.git
cd HH-Copilot/hh-extension/hh-auto-respond-extension
```

### Загрузка в Chrome

Шаг 1. Откройте Chrome и перейдите по адресу chrome://extensions.

Шаг 2. В правом верхнем углу переключите тумблер "Режим разработчика" (Developer mode) в положение ВКЛ.

Шаг 3. Нажмите кнопку "Загрузить распакованное расширение" (Load unpacked).

Шаг 4. В открывшемся диалоге выберите папку hh-auto-respond-extension (ту, что содержит manifest.json).

Шаг 5. Расширение появится в списке с названием "HH Auto-Respond" и версией 1.5.2. Убедитесь, что тумблер включён.

### Проверка работоспособности

После загрузки расширения откройте https://hh.ru и авторизуйтесь (если ещё не авторизованы). На правом краю экрана появится плавающая круглая кнопка (FAB). Через 1-2 секунды она должна изменить цвет с серого на синий, что означает успешное определение авторизации. Если кнопка стала красной -- авторизация не обнаружена, проверьте что вы вошли на сайт.

Перейдите на страницу поиска вакансий: https://hh.ru/search/vacancy?text=Python. Нажмите FAB. Откроется боковая панель с списком найденных вакансий. Каждая карточка содержит название позиции, компанию, зарплату, локацию и кнопку "Откликнуться".

Для проверки парсинга резюме перейдите на страницу одного из своих резюме (https://hh.ru/resume/{hash}). Откройте DevTools (F12), перейдите на вкладку Console. В фильтре введите HH-AR. Вы увидите логи парсинга с подробной информацией о найденных и отсутствующих полях. В sidebar откройте вкладку "Моё резюме" -- там отображаются все распарсенные данные.

### Отладка

Content script логирует все действия с префиксом [HH-AR]. Основные модули: Parser, Resume, Panel, Auth. Для диагностики DOM можно вызвать функцию diagnoseResumeDOM() из консоли (также доступна через кнопку в sidebar). Эта функция выводит полный дамп всех data-qa атрибутов на странице, что полезно при обновлении вёрстки hh.ru.

Service Worker можно проверить на странице chrome://extensions -- найдите расширение, нажмите "Service Worker" под его названием. Storage можно просмотреть, нажав "Storage" на карточке расширения.


## 4. Архитектура

### Общая схема

Расширение построено по классической архитектуре Chrome Extension Manifest V3 и состоит из трёх исполняемых контекстов: Content Script, Service Worker и Popup. Каждый контекст выполняет свою роль и взаимодействует с другими через chrome.storage.local и chrome.runtime.sendMessage.

**Content Script (content.js, 1636 строк)** -- это основной модуль расширения, загружаемый на все страницы hh.ru. Он объединяет в себе библиотечные функции (селекторы, антигаллюцинация, хранилище, тайминг, rate limiter), функциональные модули (парсер вакансий, парсер резюме, панель, авторизация) и точку входа (boot). Все модули объединены в один файл, потому что Manifest V3 не поддерживает ES modules в content_scripts -- директива import/export недоступна.

**Service Worker (background/index.js, 130 строк)** -- фоновый сценарий, работающий в контексте расширения (не на странице hh.ru). Отвечает за инициализацию хранилища при первой установке, создание ежедневного будильника (chrome.alarms) для сброса лимитов в полночь, маршрутизацию сообщений между popup и content scripts, обновление бейджа (цифры на иконке расширения).

**Popup (popup/index.html + popup/popup.js, 375 + 129 строк)** -- пользовательский интерфейс, открывающийся при клике на иконку расширения. Содержит четыре вкладки: статистика, настройки, шаблоны, логи. Данные читаются напрямую из chrome.storage.local, изменения настроек записываются туда же и передаются в content script через sendMessage.

### Потоки данных

**Поток парсинга вакансий:**

Страница hh.ru/search/vacancy загружается в браузере. Manifest V3 инжектит content.js в момент document_idle. Точка входа определяет тип страницы (search, vacancy, resume, resume-list) и запускает соответствующую логику. Для страницы поиска вызывается parseVacanciesFromPage(). Функция через findAllElements() находит все карточки вакансий, для каждой карточки извлекает данные через findElement() и safeGetText(). Результаты валидируются через validateVacancyData(), фильтруются по списку откликнутых и чёрного списка, затем передаются в renderSidebarContent() для отображения.

**Поток парсинга резюме:**

На странице /resume/{hash} вызывается parseResume(). Функция последовательно извлекает данные из DOM, используя data-qa селекторы. Для каждого поля есть основная стратегия и один или несколько fallback-вариантов. Результат сохраняется в chrome.storage.local (ключ resume) и отображается во вкладке "Моё резюме" sidebar.

**Поток авторизации:**

checkAuth() вызывается при загрузке страницы и каждые 2 секунды. Перебирает 13 селекторов и проверяет cookies. Результат обновляет состояние panelState.isLoggedIn, что меняет цвет FAB и содержимое sidebar.

**Поток сообщений:**

Popup отправляет сообщения через chrome.runtime.sendMessage (типы: get-stats, get-settings, apply-vacancy, log, settings-updated). Service Worker маршрутизирует сообщения: get-stats и get-settings читают данные из хранилища и отвечают popup, apply-vacancy пересылается в content script активной вкладки hh.ru, log записывает в хранилище.


## 5. Парсинг резюме

### Принцип работы

hh.ru использует Magritte -- собственную CSS-in-JS дизайн-систему, которая хэширует имена классов при каждом деплое. Класс вида magritte-card_bhGKz_8-5-13 изменится на magritte-card_xYzAb_9-6-14 после следующего релиза. По этой причине CSS-селекторы по классам для Magritte-компонентов не работают -- они нестабильны и ломаются при каждом обновлении сайта.

Единственный стабильный API для обращения к DOM-элементам hh.ru -- это атрибуты data-qa. Они созданы разработчиками hh.ru для внутреннего тестирования и не меняются между деплоями. Стратегия парсинга резюме строится исключительно на data-qa атрибутах, а там где они отсутствуют -- на текстовом анализе содержимого и BEM-классах библиотеки Bloko (бесхэшовые).

### Извлекаемые поля (12 из 12)

Поле "Должность" извлекается из data-qa="resume-block-title-position". Fallback -- тег h1 на странице. Это самое надёжное поле, присутствует на всех резюме.

Поле "Зарплата" берётся из data-qa="resume-block-salary". Fallback -- любой элемент с data-qa, содержащим "salary". Может быть пустым, если соискатель не указал желаемый доход.

Поля "Пол", "Возраст", "Город" извлекаются из текстового содержимого блока data-qa="resume-position-card". Magritte не предоставляет отдельных data-qa для этих полей, поэтому используется паттерн-матчинг: пол ищется по словам "мужчина"/"женщина", возраст -- по паттерну "N лет/года", город -- по текстовым фрагментам с кириллицей, не совпадающим с зарплатой или должностью.

Поле "Навыки" берётся из контейнера data-qa="skills-card". Теги навыков имеют data-qa="skill-tag-*", уровни владения -- data-qa="skill-level-title-N" (где N -- 1, 2 или 3, соответствующие "Начальный", "Средний", "Продвинутый"). Fallback -- классы .bloko-tag__text внутри skills-card.

Поле "Опыт работы" извлекается из data-qa="resume-list-card-experience". Каждая запись -- это data-qa="profile-experience-company-card". Внутри карточки структура cell-based: компания и длительность в cell-left-side (два элемента cell-text-content), позиция и период в magritte-stepper-step-content, описание -- остаточный текст после вычитания позиции и периода.

Поле "Образование" извлекается из data-qa="resume-list-card-education". Используются три fallback-стратегии: cell-left-side (как в опыте), прямые дочерние элементы eduCard, полный текстовый скан с поиском паттернов "Название -- год". UI-текст фильтруется (слова "Образование", "Редактировать" и т.д.).

Поле "Языки" ищется в bloko-tag внутри data-qa="resume-about-card" и data-qa="resume-position-card". Отдельного data-qa для языков на текущей версии Magritte нет.

Поле "Дополнительная информация" берётся из data-qa="resume-about-card" -- текстовый контент всего блока.

### Диагностика

Для подбора селекторов при изменении DOM hh.ru предназначена функция diagnoseResumeDOM(). Она доступна из консоли (window.__hhDiagnose) или через кнопку в sidebar. Функция собирает все data-qa атрибуты на странице, группирует их по префиксу, проверяет все resume-селекторы из HH_SELECTORS, выводит дампы секций опыта и образования, показывает заголовки h1-h3 и все секции страницы.

## 7. Технологии

### Manifest V3

Расширение построено на Manifest V3 -- текущей версии платформы Chrome Extensions. Manifest V3 заменил V2 в 2023-2024 годах и внёс ряд существенных ограничений: content_scripts не поддерживают ES modules (import/export недоступны), фоновые страницы заменены на Service Workers (без доступа к DOM), удалены API вроде chrome.webRequest для блокировки запросов, данные хранятся исключительно через chrome.storage.local/session. Все решения в проекте принимались с учётом этих ограничений. Отсутствие модульной системы привело к объединению всех библиотек в один файл content.js. Service Worker не имеет доступа к DOM и работает только через chrome API.

### Shadow DOM

Панель расширения (sidebar) создаётся внутри Shadow DOM с параметром mode: "closed". Это означает полную изоляцию: CSS-стили hh.ru (библиотеки Bloko и Magritte) не проникают внутрь панели, а стили панели не влияют на страницу. JavaScript панели изолирован от глобального контекста страницы -- нет конфликтов переменных, нет переопределения прототипов. Shadow DOM -- это встроенный механизм браузера, не требующий полифилов или библиотек. Он работает во всех современных браузерах на базе Chromium, включая Chrome, Edge, Brave, Opera.

### chrome.storage.local

Все данные расширения хранятся в chrome.storage.local -- ключевом хранилище, доступном из всех контекстов расширения (content scripts, service worker, popup). Хранилище асинхронное, объём ограничен 10 МБ на расширение (достаточно для тысяч вакансий). Схема хранения: settings (объект с настройками пользователя), stats (статистика откликов), appliedVacancies (массив id откликнутых вакансий), skippedVacancies (пропущенные), blacklistedCompanies (чёрный список компаний), logs (массив до 500 записей), resume (объект с распарсенным резюме), resumeList (список резюме), dailyResetDate (дата последнего сброса), installedAt (дата установки). Данные не отправляются на внешние серверы.

### data-qa селекторы

Стратегия обращения к DOM hh.ru строится исключительно на атрибутах data-qa. Каждый селектор в HH_SELECTORS -- это массив строк, перебираемых по порядку приоритета. Первый элемент -- наиболее стабильный data-qa, последующие -- fallback-варианты (BEM-классы Bloko, partial class match). Функция findElement() перебирает массив, проверяет существование, принадлежность к документу, видимость (display и visibility через getComputedStyle) и возвращает первый подходящий элемент. Magritte-хэшированные классы (вида magritte-card_bhGKz) категорически не используются -- они нестабильны.

### React-safe input

hh.ru построен на React 18 с Magritte UI. React использует synthetic events, поэтому прямое присвоение element.value = text не обновляет внутреннее состояние React, и данные не отправляются при сабмите. Расширение использует native value setter (Object.getOwnPropertyDescriptor для HTMLTextAreaElement.prototype.value) с последующей диспетчеризацией synthetic событий (input и change, bubbles: true). Это гарантирует, что React "увидит" новое значение и корректно его обработает.


## 8. Версионирование

### Правила

Версия расширения указывается в manifest.json в поле "version" и следует формату SemVer (MAJOR.MINOR.PATCH). Каждый коммит в основную ветку с функциональными изменениями обязан сопровождаться бампом версии. Правила бампа:

**PATCH (третий разряд).** Исправление багов, обновление селекторов без изменения логики, мелкие правки стилей popup, обновление текста документации. Пример: 1.5.0 переходит в 1.5.1.

**MINOR (второй разряд).** Новый функционал, не ломающий существующее поведение: добавление нового парсера, новые поля в резюме, новые настройки в popup, новые вкладки в sidebar. Пример: 1.5.0 переходит в 1.6.0.

**MAJOR (первый разряд).** Изменения, ломающие обратную совместимость: новое архитектурное решение, миграция схемы хранилища, удаление существующего API. Пример: 1.5.0 переходит в 2.0.0.

### Формат коммитов

Каждый коммит должен начинться с типа изменения: feat:, fix:, refactor:, docs:, chore:. После типа -- краткое описание на русском или английском. Примеры: "feat: парсинг навыков с уровнями", "fix: offsetParent для fixed элементов", "docs: обновление ARCHITECTURE.md".

### История изменений

Полная история версий ведётся в файле CHANGELOG.md в корне папки расширения. Формат -- Keep a Changelog. Каждый релиз содержит секции "Добавлено", "Изменено", "Исправлено", "Удалено". Текущая версия -- 1.5.0. Предыдущие: 1.3.0 (переписан парсинг резюме под Magritte), 1.2.0 (исправлен баг загрузки резюме), 1.1.0 (парсер резюме, табовая система sidebar), 1.0.0 (базовая архитектура).


## 9. Разработка

### Как добавить новый функционал

Процесс добавления нового функционала в расширение состоит из нескольких этапов. Сначала необходимо определить, в каком контексте будет работать новый код. Если функционал связан с DOM hh.ru -- код добавляется в content.js. Если это фоновая задача или работа с chrome API -- в background/index.js. Если это UI для popup -- в popup/папку.

**Добавление нового селектора.** Откройте объект HH_SELECTORS в начале content.js. Найдите соответствующую группу (Vacancy Search, Vacancy Page, Resume Page, Auth) или создайте новую. Добавьте селектор в массив: первым элементом -- data-qa (самый стабильный), затем fallback-варианты. Протестируйте через diagnoseResumeDOM() -- она проверит все resume-селекторы и покажет, какие найдены, а какие нет.

**Добавление нового модуля в content.js.** Все модули в content.js организованы секциями с комментариями-разделителями. Новый модуль должен быть добавлен после библиотечных модулей (LIB) и перед точкой входа (MAIN/BOOT). Обязательно используйте createLogger('ModuleName') для логирования. Все DOM-операции должны идти через findElement/safeGetText/safeClick, никогда не через прямой querySelector.

**Добавление нового ключа в хранилище.** Если новый функционал требует хранения данных, добавьте ключ в инициализацию в background/index.js (в секции chrome.runtime.onInstalled.addListener) и в обёртку storage в content.js. Обязательно предусмотрите значение по умолчанию.

### Правила кода

Правило 1. Ничего не возвращай как undefined. Все функции возвращают string, null, boolean, number или object. Пустая строка '' -- допустимое значение, undefined -- нет. Это предотвращает cascading ошибки при цепочках вызовов.

Правило 2. Никогда не предполагай, что DOM-элемент существует. Любой вызов querySelector может вернуть null. Используй findElement() или safeQuery() вместо прямого querySelector.

Правило 3. Никогда не обращайся к textContent/value без обёртки. safeGetText() проверяет существование, тип, видимость и пустоту. safeGetAttr() проверяет существование и возвращает fallback. safeInput() проверяет disabled/readonly и использует React-safe setter.

Правило 4. Никогда не кликай по невидимым элементам. safeClick() проверяет offsetParent, getComputedStyle, disabled. Если элемент невидим -- функция возвращает false без выполнения клика.

Правило 5. Всегда валидируй данные перед использованием. validateVacancyData() проверяет тип, длину, формат. ExtractVacancyId() использует regex с fallback на пустую строку.

Правило 6. Не используй Magritte-хэшированные CSS-классы. Только data-qa атрибуты и Bloko BEM-классы. Хэши меняются при каждом деплое hh.ru.

Правило 7. Все стили панели должны быть внутри Shadow DOM. Никаких инлайн-стилей на элементах вне shadowRoot. Никаких вставок в document.head.

Правило 8. Логируй всё, что может быть полезно для отладки. Префикс [HH-AR][ModuleName]. Уровни: debug (info), warn, error.


## 10. Roadmap

### Фаза 1 (завершена): Базовая структура

Фаза 1 включала создание всей инфраструктуры расширения: конфигурация Manifest V3, единый бандл content.js, Service Worker, popup с 4 вкладками, библиотека селекторов (40+ групп), антигаллюцинационная система (safe-функции, валидация, верификация действий), обёртка chrome.storage.local с дефолтными значениями, rate limiter с адаптивным замедлением, парсер вакансий со страницы поиска, авторизационный gate (13 селекторов + cookie fallback), SPA-observer для MutationObserver, FAB-кнопка с тремя состояниями, Shadow DOM sidebar с табами. Также в рамках фазы 1 был реализован полный парсер резюме (12 полей), список резюме (/applicant/resumes), контекстно-зависимая кнопка загрузки, DOM-диагностика. Все задачи фазы 1 выполнены.

### Фаза 2: Интеллектуальный мэтчинг и отклик

Планируется реализация matching-движока для вычисления релевантности вакансий резюме пользователя. Пятикомпонентный скоринг учитывает навыки (Jaccard similarity с alias matching: k8s = kubernetes, pg = postgresql), зарплату (overlap-based с 30% tolerance), опыт (range parsing), название позиции (word overlap + keyword boosting), локацию (exact/remote matching). Результат скоринга -- число от 0 до 100, отображаемое цветовой кодировкой в sidebar.

Дальнейший подфазовый план: детальный парсер вакансии (переход на /vacancy/{id}, извлечение описания, требований, навыков из текста), восьмишаговый процесс отклика (проверка лимитов, навигация, поиск кнопки, верификация, клик, ожидание popup, обработка алертов, заполнение письма, отправка), AI-генерация сопроводительных писем (промпты подготовлены в docs/reference-prompts.py), массовый auto-apply с сортировкой по match score.

### Фаза 3: Переговоры и аналитика

Планируется трекер переговоров: отслеживание новых сообщений от работодателей в разделе /applicant/negotiations, автоответ по шаблонам, уведомления в popup. Аналитический модуль: статистика откликов по дням/неделям, skill gap анализ (какие навыки востребованы в вакансиях с высоким match score, но отсутствуют в резюме), рекомендации по улучшению резюме.

### Фаза 4: Распространение

Публикация в Chrome Web Store, порт на Firefox Add-ons (Manifest V2 адаптация, поскольку Firefox не полностью поддерживает V3), порт на Edge Add-ons (минимальные изменения -- Chromium-база). Возможна реализация автообновления селекторов через внешний сервис при обнаружении несовпадений DOM.
